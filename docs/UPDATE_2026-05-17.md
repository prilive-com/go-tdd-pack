# Update — 2026-05-17 (tool grounding + confidence + tighter orientation)

Use this if your project already installed v2.0 from
`docs/V2_ROLLOUT_GUIDE.md` (commit `db23cec` or earlier) and you want
to pull in the quality tuning landed in commits `ced4e38` and
`12eceb8`.

If you're installing v2.0 for the first time, read
`V2_ROLLOUT_GUIDE.md` instead — it already includes everything here.

---

## What this update adds

1. **Tool grounding** — the runner pre-executes `go vet`, `gofmt -l`,
   `staticcheck`, `golangci-lint`, `govulncheck` and feeds their output
   to Codex with the diff. Codex gets deterministic tool evidence
   instead of re-deriving the same signal (or missing it). Skips
   silently if tools aren't installed; skips entirely if no `go.mod`.

2. **Confidence scores on every finding** (1-5) — required schema
   field. Claude sees `[blocker/correctness c=4]` instead of just
   `[blocker/correctness]`, so it can triage by certainty as well as
   severity.

3. **Tighter context orientation** — Codex now sees only
   `git diff --name-only HEAD` for orientation, not the full file
   tree. Less attention dilution. (Codex still has full repo access
   via tools.)

4. **Quality-tuned config** — `max_rounds: 4→5`,
   `reasoning_effort: high→xhigh`, `min_surface: minor→nit`,
   thoroughness instructions added to the system prompt and round-N
   prompt.

## What changes in observable behavior

After this update, Codex's round-1 output will look richer in three
visible ways:

**1. Tool grounding block at the top of every cycle's prompt:**

```
## Tool grounding (pre-executed before this review)

### gofmt -l ./...
(clean)

### go vet ./...
(clean)

### staticcheck ./...
internal/foo/bar.go:42:3: ineffective assignment to t (SA4006)

### golangci-lint run
...

### govulncheck ./...
...
```

If your project doesn't have these tools installed, each line will
show `(skipped: <tool> not installed)`. Install them for full benefit.

**2. Confidence scores in injected findings:**

Before:
```
- [blocker/correctness] Potential nil dereference
  ...
```

After:
```
- [blocker/correctness c=4] Potential nil dereference
  ...
```

`c=N` reads: 1=guess, 2=plausible, 3=likely, 4=high (read surrounding
code), 5=verified (ran the tool/test or cited a doc).

**3. Changed-files block instead of full repo tree:**

Before: Codex saw `git ls-files` output (all paths).
After: Codex sees only the paths that appear in the diff.

If you observe Codex asking for files outside the changeset more
often, that's expected — and fine. Codex has `git ls-files` and
`Read` available; it pulls when needed.

## What the AI doing the coding should now do differently

Mostly nothing — the cycle still runs in the background, findings
still inject the same way, you still fix silently or push back.

Two small behavioral shifts:

**Use confidence for triage.** When a `[blocker c=1]` and a
`[minor c=5]` arrive together, the high-confidence minor is more
worth addressing immediately than the speculative blocker. Push back
on low-confidence blockers if you have grounds.

**Trust tool-grounded findings more.** When a finding cites a
specific `staticcheck` or `go vet` line, the tool already verified it
— don't argue with it; fix it. When a finding has no tool citation,
normal disagreement protocol applies.

## Rollback

If anything breaks:

```bash
git -C ~/myproject revert <commit-of-this-update>
```

Or restore the previous versions from your last working starter
checkout. None of the schema/prompt changes are runtime-breaking —
older Codex outputs (without `confidence`) will fail schema validation
on round 1, but the runner records `failed:invalid_json` rather than
exploding. Once rolled back, the next cycle will succeed normally.
