# Go Testing Rules

Tests are production code.

## What to test

- Every production bugfix ships with a regression test in the same change.
- Every new public API or handler has at least one test covering its
  happy path plus at least one failure mode.
- Every change to initialization, startup, restart, or reconciliation
  logic explains the restart scenario and covers it with a test.
- Every parser or input-handling change adds or updates fuzz/property
  coverage.
- Every change touching concurrency (goroutines, channels, mutexes,
  workers) considers race coverage.

## Required patterns

- Tests prove behavior, not implementation details.
- A test must be able to fail. Tests that only `t.Log` are not valid.
- Real assertions (`require.Equal`, `assert.Equal`), not just absence
  of error.
- Edge cases, failure paths, cancellation/timeouts, concurrency paths
  covered.
- Table-driven tests with >=3 rows or collapse to non-table form.
- `t.Helper()` in test helpers so failure messages point at the caller.
- `t.Cleanup(func(){...})` over `defer`. Runs in reverse order of
  registration and survives `t.Parallel`.
- `t.Parallel()` for leaf tests that don't share mutable state.

## Patterns

- `embed-nil-interface` fakes for cross-module dependencies.
- Compile-time interface assertions: `var _ Iface = (*T)(nil)`.
- `t.TempDir()` for filesystem work.
- `t.Setenv` when testing env-var-reading code (auto-restores).
- Fixed seeds for randomness; `testing/cryptotest.SetGlobalRandom` for
  crypto tests.
- `testing/synctest` (Go 1.25+) instead of `time.Sleep` for time-sensitive
  tests.
- `t.ArtifactDir()` (Go 1.26+) for artifacts to preserve.
- `t.Context()` (Go 1.24+) for test contexts; otherwise
  `context.WithTimeout(context.Background(), N*time.Second)`.

## Forbidden

- Mocking the system under test or same-package collaborators.
- Mocking `pgx.Conn`, `http.Client`, etc. directly. Wrap them in your
  own interface first.
- Tautological assertions (`assert.NoError(err)` with no follow-up).
- Snapshot/golden files generated by the same agent that wrote the
  code, without justification.
- Sleep-based synchronization unless absolutely unavoidable.
- Skipping a failing test with `t.Skip`. If it's broken, fix it or
  delete it. Skipped tests rot.
- Weakening a production assertion to make a test pass.
- Deleting a test to make a build green without explicit reasoning in
  the commit message.

## When a test fails during your work

1. Determine whether the test reveals a real defect or is merely brittle.
2. If real: surface it. The failure is the signal.
3. If brittle: fix the brittleness (remove ordering assumptions, loosen
   timing bounds, stub out nondeterministic dependencies).
4. Never weaken a production assertion to make a test pass.
5. Never delete a test to make a build green without explicit reasoning
   in the commit message.

## How to test

- Prefer integration-style tests that exercise real code paths with fake
  inputs over unit tests that mock everything.
- Prefer real databases (testcontainers, ephemeral Postgres) over DB
  mocks when the code-under-test involves SQL.
- Prefer property-based tests for functions with clear invariants.
- Table-driven tests are the default for functions with multiple input
  variants.

## Library projects

- `Example_xxx` tests for exported APIs.
- Consumer-perspective tests in `package_test`.

## Suggested commands

- `go test ./...`
- `go test -race ./...`
- `go test -race -count=1 ./...`
- `go test -run TestName -count=100` for flake-prone tests
- `go test -fuzz=FuzzName -fuzztime=30s` for parsers/validators

## When in doubt

Propose the smallest missing proof, not the biggest test suite. One
targeted regression test is worth more than a full coverage sweep.
