# REVIEW.md — Go Code Review Rules

You are a Staff+ Go reviewer. This file overrides default review
behavior for this repository.

## What "Important" means here

Reserve **Important** for findings that would cause:

- crash, data loss, deadlock, data race
- security vulnerability, auth/authZ bypass, credential/PII exposure
- goroutine/resource leak under realistic conditions
- accounting/money invariant break
- broken rollback or migration path
- backward-incompatible public API break (libraries)
- hallucinated package import (slopsquatting risk)
- TDD ceremony bypass on a Tier 1 high-stakes path

Style, naming, refactoring, and modernization suggestions are **Nit at
most** unless tied to a specific failure mode above.

## Cap the nits

Report at most 5 Nits per review. If more exist, aggregate as "plus N
similar items" in the summary.

If everything you found is Nit, lead with "No blocking issues."

## Do not report

- Anything CI already enforces: lint, formatting, type errors
- Generated files (`*.gen.go`, `*.pb.go`, `*_mock.go`)
- Test-only code that intentionally violates production rules

## Always check

1. **TDD ceremony adherence.** If the diff touches a path matched by
   `.tdd/tdd-config.json` `tier1_path_regexes`:
   - Was there a corresponding `red(<id>):` commit before the
     `green(<id>):` commit?
   - Is `.tdd/current-plan.md` present and complete with all required
     markers `yes`?
   - Is there a `red-proof.md` artifact with verbatim test output?
   - Does the red-proof include "Why this is not a false red"?
   - If any answer is no, flag as **Important** ("TDD ceremony
     bypassed").

2. **Necessity audit (AI-bloat).** Every new exported symbol, file,
   interface, dependency, abstraction:
   - >=2 concrete callers in this PR
   - Or part of documented public library API
   - Or required by a business invariant
   Otherwise flag as **Nit** with a deletion recommendation.

3. **Supply chain.** Every new `go.mod` `require` line:
   - Must be on `.claude/allowed-modules.txt` (or PR description must
     justify)
   - Must resolve to a real, maintained package on pkg.go.dev
   - Must pass `govulncheck`

4. **Money/decimal handling.** Any new code touching
   money/prices/balances/quantities:
   - Must NOT use `float64`
   - Must preserve precision through JSON/DB round-trips

5. **Goroutine lifecycle.** Every `go` statement must have a documented
   termination path.

6. **Test integrity.** Tests must be able to fail; mocks model real
   dependencies; no tautological assertions.

## Severity escalation for AI-authored PRs

If the PR is AI-authored (commit message, `Co-authored-by:` LLM trailer,
`// generated by` markers, or implausibly broad scope), review at one
severity tier higher. Hallucinated imports → Important; tautological
test that masks a bug path → Important.

## Required output structure

1. Risk map of changed files
2. Executive summary (2–3 sentences, project type, decision)
3. Business/domain invariants the PR must preserve
4. Important findings (file:line, code quote ≤10 lines, failure
   scenario, minimal fix, tests to add)
5. Should-fix findings (P2)
6. Nits (max 5)
7. Necessity / AI-bloat audit table
8. TDD ceremony check (if applicable)
9. Test evidence (what existing tests prove and don't prove)
10. Release / compatibility notes
11. Follow-up tickets

## Format

- Cite file:line for every finding.
- Quote ≤10 lines of code per finding.
- Do not invent evidence. If you can't verify, mark "Needs more
  context" with what's missing.
- Do not claim commands were run unless you ran them.
